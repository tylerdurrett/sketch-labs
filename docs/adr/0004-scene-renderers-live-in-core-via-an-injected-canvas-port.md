# Scene Renderers live in core, injected via a core-defined Canvas2D port

A **Scene Renderer** (the backend that turns a **Scene** into pixels/SVG) lives in `packages/core`, not in a consumer, because CONTEXT.md's definition is that *one* Scene Renderer serves every vector Sketch, and the consumers are real: the studio (#6), Remotion (#11, whose acceptance criterion is frame-for-frame parity with the studio), and SVG export (#9). A studio-side adapter would leave `packages/video` unable to import it (apps sit atop the dependency graph), forcing either a second hand-maintained Canvas2D implementation â€” putting #11's parity at risk â€” or a later hoist into core anyway, with more churn.

The Canvas2D Scene Renderer does **not** depend on the global `CanvasRenderingContext2D` type. core's tsconfig is deliberately `lib: ["ES2022"]` with no `"DOM"` â€” the type-level guardrail that keeps core headless and underwrites the determinism proof (#27/#30). Adding `"DOM"` to satisfy the renderer would reopen the entire `window`/`document`/`createElement` surface to core and let a future contributor write real DOM code there with a green typecheck. Instead core declares a **minimal structural port** â€” a `Canvas2DContext` interface naming only the canvas methods the renderer calls. The real browser `CanvasRenderingContext2D` is structurally assignable to it, so the studio and Remotion pass `canvas.getContext('2d')` directly with no adapter, while core's lib stays `["ES2022"]` and the headless boundary stays bright-line.

## Consequences

- The renderer is a pure context-injected function â€” `renderToCanvas(ctx: Canvas2DContext, scene: Scene): void` â€” with no DOM imports. The caller (studio, Remotion frame, or a node test passing a stub/`node-canvas` context) owns surface creation, sizing, and DPR; core owns only drawing the Scene's Primitives in painter's order.
- The `Canvas2DContext` port lists exactly the canvas API surface the renderer touches, and can widen (never rework) as renderers grow. core's `lib: ["ES2022"]` must stay free of `"DOM"`; reaching for `CanvasRenderingContext2D` or any `lib.dom` global in core is the anti-pattern this ADR exists to prevent.
- The coordinate-space â†’ pixel mapping (fit/letterbox/DPR) is a caller concern, layered over this function, not baked into it â€” the renderer draws in the Scene's coordinate space and the caller establishes the transform.

## Amendment (issue #11): the fit mapping graduates to a shared core helper

The original decision left the coordinate-space â†’ pixel *mapping* in each caller. Issue #11 (Remotion parity) showed that leaving it there makes cross-caller frame-for-frame parity a coincidence rather than a guarantee: the studio and Remotion would each carry their own fit-and-draw glue, free to drift, and a parity test could only ever prove "Remotion matches a copy of the pipeline the studio doesn't run." So the mapping graduates into core as two shared pieces: `computeContainFit(...)` (the pure fit math, moved out of the studio) and `drawSceneFitted(ctx, scene, pixelW, pixelH)` (compute fit â†’ `setTransform` â†’ `renderToCanvas`). Both the studio's `LiveCanvas` and the Remotion composition call `drawSceneFitted`, so parity is *structural* â€” one function, not two lookalikes â€” and the cost is that this fit convenience now lives in core where a reader might expect (per the consequence above) a caller concern.

The core invariants are unchanged: `renderToCanvas` still establishes no transform of its own (`drawSceneFitted` is a *separate* function layered over it), and **DPR remains a caller concern** â€” each caller sizes its own backing store and hands `drawSceneFitted` the final pixel dimensions (the studio passes `canvas.width` = `css Ă— dpr`; Remotion passes the composition width). What moved into core is the deterministic fit/letterbox math and its application, precisely because that is the part that must be *identical* across callers for parity to hold; what stays with the caller (surface creation, sizing, DPR, clearing) is the part that legitimately differs.

## Amendment (issue #92): clearing/background graduates too, as a Render Setting

Two linked corrections to the boundary this ADR governs.

**Clearing graduates into `drawSceneFitted`.** The previous amendment listed "clearing" among the parts that "legitimately differ" and stay with the caller. That was wrong for the same reason the fit mapping was: parity is the driving invariant â€” *what you see == what you get* â€” and it is unattainable if each caller paints (or fails to paint) its own backdrop. The shared pipeline had no backdrop at all, so a black-stroked Sketch was invisible where no host page sat behind the canvas, the alpha-less `.mp4` flattened transparent â†’ black into an all-black frame, and the studio/SVG only looked right because a CSS/page background happened to show through â€” a parity break masked by three different accidents. `drawSceneFitted` now resets to identity and, *before* the fit, paints (or, for `'transparent'`, clears) the **full pixel surface** (letterbox included) unconditionally. That single unconditional clear/fill also subsumes the per-frame surface clear (fixing a missing `clearRect` that had left cross-frame ghosting latent in the Remotion loop), so callers no longer clear at all. **DPR and surface sizing remain caller concerns** â€” only clearing/background moved.

**Background is a Render Setting, not Scene IR.** The backdrop is resolved and validated in the caller layer (`resolveRenderSettings`, alongside `fps`/`width`/`height`) with a safe opaque-white default, and threaded into `drawSceneFitted` â€” it is NOT Scene IR content (`{space, primitives}` is unchanged) and NOT per-Sketch logic, so no Sketch author has to think about it and every caller inherits a safe default with zero discipline. The SVG serializer mirrors it as a full-`viewBox` `<rect>` at the bottom of the z-order so SVG == raster. This deliberately does NOT introduce a Scene-authored background *layer* (out of scope). A new ADR number is not warranted: this amends the caller/core boundary ADR-0004 already governs, it does not overturn it.
