# Stateful Sketches are Harness-driven fixed-timestep folds

A simulation **Sketch** (particles, flocking, anything whose state at frame N depends on frame N−1) does not violate [ADR-0002](0002-sketch-is-a-pure-function-of-params-seed-time.md). The invariant it protects is that the **Scene** at time `t` is fully determined by `(params, seed, t)` — *not* that no state exists. A stateful Sketch satisfies the invariant by being a deterministic **fold**: the author exports `initial(params, seed)`, a fixed-`dt` `step(state, …) → state`, and a `draw(state) → Scene`, and the **Harness** owns the loop, computing `generate(params, seed, t)` as `draw(fold(step, initial, ⌊t / dt⌋))`. The author never holds a `requestAnimationFrame`-driven mutable array across calls — that remains the anti-pattern ADR-0002 forbids.

We chose a **fixed** timestep (not wall-clock `dt`) because reproducibility requires that the number and size of steps taken to reach `t` be a function of `t` alone; a variable `dt` would fold a different step count on a fast vs. slow machine and produce different Scenes for the same `(params, seed, t)`, breaking deterministic video and frame-exact plotter export. The simulation timestep is decoupled from the display rate: exploration drives `t` from wall-clock and the Harness folds the fixed steps needed to reach it, so a sim plays at the *same speed* on every machine — faster machines merely render more frames and interpolate between bracketing steps for smoother motion.

## Consequences

- **Step and draw are separate, and run at different rates.** `step` is domain → domain and may run many times per displayed frame (substepping for stability); `draw` is domain → Scene and runs only for frames actually shown or exported. Collapsing them would rebuild Primitives on every substep.
- **The Harness owns caching, and caching is invisible to output.** Sequential playback folds forward incrementally (O(t), not O(t²)); random access (scrub) resumes from the nearest checkpoint. The cache is keyed on `(params, seed)` and invalidated when they change — a cache bug can only change *speed*, never the frame. Precomputing and persisting the whole trajectory this way is **Bake** (the Blender sense).
- **Heavy sims degrade, they don't desync.** If a machine cannot fold the fixed steps in real-time, the Harness clamps steps-per-frame (graceful slow-mo) or the sim is baked first — never a variable `dt` that would sacrifice determinism.
- **Interpolation between steps is a display-only nicety.** It smooths motion for fast renderers; the canonical exported frame is the folded state itself, so interpolation never affects Remotion or plotter output.
