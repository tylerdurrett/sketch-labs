# ColorParamSpec joins the ParamSpec union; Randomize skips colors

`ParamSpec` — the open, `kind`-discriminated union behind every **Parameter Schema** — gains its first non-numeric member: `ColorParamSpec = { kind: 'color'; default: string }`. This is the union's first widening since `NumberParamSpec` was frozen (issue #47), driven by leaf-field's `backgroundColor`/`discColor` knobs, and it exercises the "purely additive" promise the union was designed around: the control panel grows a `case 'color'`, `defaultParams` and the Preset shape need nothing (both are already kind-generic), and the numeric member is untouched.

**The value domain is a canonical hex CSS color string** (`'#1a2b3c'`), not the full CSS color grammar. The Studio-owned picker converts its saturation/value field, hue slider, and integer RGB fields to and from the 7-character `#rrggbb` form. Palette swatches are Harness-wide editing shortcuts, currently fixed to black and white, and emit that same representation; they are not Presets or additions to the Parameter Schema. One representation keeps stored values, equality, Preset round-trips, reproduction metadata, and both renderers (Canvas2D `fillStyle`, SVG `fill`) aligned. We considered accepting any CSS color string: rejected because two spellings of one color (`'white'` / `'#ffffff'`) would compare unequal on the determinism spine.

**Randomize passes colors through, so color controls have no Lock affordance.** `randomize` rolls only `kind: 'number'` specs; a color param survives every roll unchanged. A color is a deliberate aesthetic choice: there is no meaningful "uniform roll" over hex the way there is over a numeric `[min, max]`, and a random `#xxxxxx` is almost always garbage next to a chosen palette. Lock remains Randomize-exclusion for numeric controls only. Showing it for a color would imply an effect it cannot have, so the color row omits it. If colors should ever roll, that is a palette mechanism deserving its own decision, not a uniform sample.

**Presets need no migration — including generic color lock keys.** Preset reconciliation (ADR-0002 spine, `applyPreset`) is presence-only and kind-agnostic: schema keys absent from a stored Preset fill from their spec `default`, unknown keys drop, stored values load as-is, and schema-present entries in the generic `locks` array remain intact. A legacy or externally-authored color lock therefore reloads into the Studio's `Set<string>` and serializes again unchanged. It has no UI or Randomize effect, but filtering it would be an unnecessary Preset rewrite, so `PRESET_VERSION` and the Preset shape do not change. The corollary for params remains: an absent key tracks the *current* default, so a pre-widening Preset whose image must survive a later default change pins the new color keys explicitly.

## Consequences

- The schema helpers in `sketch-util.ts` are typed over mixed schemas: `numberParam`/`colorParam` accept any `Record<string, ParamSpec>` schema and narrow their `key` to the keys of the right kind via a mapped conditional (`KeysOfKind`), so reading a color knob with `numberParam` is a compile error. Single-kind schemas (`satisfies Record<string, NumberParamSpec>`) still typecheck unchanged.
- An unknown `kind` still hits the control panel's loud fallback — `color` graduating to a real control does not soften the safety net for the kinds (boolean, enum, …) that remain deferred (CONTEXT.md).
- Randomize's color pass-through is pinned by test; a future palette-roll feature must change that contract deliberately, not by accident.
- Numeric rows retain their Lock toggle; color rows do not. Preset reload/save coverage pins existing color lock keys as preserved, inert compatibility data rather than a migration target.
