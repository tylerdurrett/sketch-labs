# ColorParamSpec joins the ParamSpec union; Randomize skips colors

`ParamSpec` — the open, `kind`-discriminated union behind every **Parameter Schema** — gains its first non-numeric member: `ColorParamSpec = { kind: 'color'; default: string }`. This is the union's first widening since `NumberParamSpec` was frozen (issue #47), driven by leaf-field's `backgroundColor`/`discColor` knobs, and it exercises the "purely additive" promise the union was designed around: the control panel grows a `case 'color'`, `defaultParams` and the Preset shape need nothing (both are already kind-generic), and the numeric member is untouched.

**The value domain is a hex CSS color string** (`'#1a2b3c'`), not the full CSS color grammar. The native `<input type="color">` the control panel binds to speaks only the 7-character `#rrggbb` form — it emits hex and renders nothing else — so pinning the param domain to hex keeps the stored value, the control, the Preset round-trip, and both renderers (Canvas2D `fillStyle`, SVG `fill` both accept hex natively) in one representation with no normalization or parsing layer. We considered accepting any CSS color string: rejected because the control would then need a CSS-color→hex converter just to display the value, and two spellings of one color (`'white'` / `'#ffffff'`) would compare unequal on the determinism spine.

**Randomize passes colors through — as a stated contract, not an implementation accident.** `randomize` rolls only `kind: 'number'` specs; a color param survives every roll unchanged, locked or not. A color is a deliberate aesthetic choice: there is no meaningful "uniform roll" over hex the way there is over a numeric `[min, max]`, and a random `#xxxxxx` is almost always garbage next to a chosen palette. So Randomize is numeric-only for now; if colors should ever roll, that is a palette mechanism deserving its own decision, not a uniform sample. The Lock toggle still renders on color controls — it excludes nothing, but uniform control chrome beats a per-kind affordance, and it stays harmless.

**Presets need no migration — but a pre-widening Preset whose image must survive a later default change pins the new keys explicitly.** Preset reconciliation (ADR-0002 spine, `applyPreset`) is presence-only and kind-agnostic: schema keys absent from a stored Preset fill from their spec `default`, unknown keys drop, stored values load as-is, so `PRESET_VERSION` stays at 1. The corollary: an absent key tracks the *current* default, so a pre-color Preset's rendered image silently follows any future default change. leaf-field's "Nice One" therefore pins `backgroundColor`/`discColor` to white explicitly — its captured white-on-white image predates the knobs, and the sketch's shipped defaults (white discs on mid gray) would otherwise repaint it on load.

## Consequences

- The schema helpers in `sketch-util.ts` are typed over mixed schemas: `numberParam`/`colorParam` accept any `Record<string, ParamSpec>` schema and narrow their `key` to the keys of the right kind via a mapped conditional (`KeysOfKind`), so reading a color knob with `numberParam` is a compile error. Single-kind schemas (`satisfies Record<string, NumberParamSpec>`) still typecheck unchanged.
- An unknown `kind` still hits the control panel's loud fallback — `color` graduating to a real control does not soften the safety net for the kinds (boolean, enum, …) that remain deferred (CONTEXT.md).
- Randomize's color pass-through is pinned by test; a future palette-roll feature must change that contract deliberately, not by accident.
