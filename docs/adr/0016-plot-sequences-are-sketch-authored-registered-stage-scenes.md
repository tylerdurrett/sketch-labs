# Plot Sequences are Sketch-authored registered Stage Scenes

A **Plot Sequence** is an optional generic Sketch capability that produces a fixed ordered set of named **Plot Stage** Scenes from one shared authored input set, Seed, time, Composition Frame, Page Frame, and physical page mapping. The Sketch—not the Harness—defines each Stage and owns whether its Scene derives from source data, the Primary Scene, or private intermediate geometry. One Stage is explicitly Primary and remains the Sketch's ordinary `generate` output, preserving existing consumers and Presets; Sequence-aware Studio views may isolate another Stage or overlay them for registered comparison.

This boundary keeps the Scene IR flat and renderer-agnostic. A Plot Stage is a complete plotter asset, not a Primitive tag, painter-order layer, embedded registered Sketch, or generic workflow node. The Harness owns shared registration, Stage selection, grouped controls, retained preparation, Page finalization, Preset persistence, and export presentation without learning the Sketch's artistic derivation policy. Every Stage receives the unchanged Sequence Seed for shared generated geometry; namespaced sub-seeds are allowed only for additional Stage-local randomness.

Stage identity is instance identity, distinct from the reusable headless **Stage generator** that produces its Scene. Runtime, preparation, and output state are keyed by the stable Stage identity even when two present or future Stages use the same generator with different controls. The first version exposes only the fixed instances authored by the owning Sketch—no Add, Duplicate, or Delete operations—but must not use generator identity as the unique Stage key. This keeps the curated workflow small while preserving a later additive path to artist-authored duplicate instances.

The owning Sketch keeps one Parameter Schema as the established Studio control, Randomize, Lock, history, and Preset spine. Sequence metadata assigns its keys to shared or Stage ownership, and the Sequence boundary projects only the resolved shared and Stage-owned values into each Stage runtime identity and generator call. Workers, caches, and retained results therefore depend on a Stage-local parameter record rather than flat owning-Sketch key names. A future composition document may persist nested per-instance parameters and feed that same runtime directly; version one does not replace the existing schema spine speculatively.

Stage preparation is demand-driven and retained. Opening Photo Scribble in its Primary Ink view starts no supporting work; selecting Watercolor Forms, entering Combined, or requesting Watercolor export prepares only the missing or stale Stage identities and reuses current retained results. This avoids making startup cost proportional to every curated Stage and lets later additions use the same scheduler without special cases.

Preparation retains the last coherent isolated or Combined snapshot while current work is visibly updating. A shared-input revision identifies compatible Stage results, so Combined never presents a newly prepared Stage beside another Stage from an older image, Composition Frame, or other shared input; the prior matching snapshot may remain until the current set is ready. Export requires the requested Stage result to match the current revision.

A Stage's registration identity and preparation identity remain distinct. The former covers the shared source and coordinate inputs that must match for registered presentation; the latter contains only the Stage generator's projected inputs and explicitly declared Seed and time participation. A new Ink Seed can therefore invalidate Ink while the unseeded Watercolor Forms result remains current and registration-compatible, without a general dependency graph or generator-specific cache exception.

A Stage-local preparation failure does not poison the Sequence. That Stage's unavailable or stale artifact remains non-exportable, while every other current Stage remains viewable, editable, and independently exportable; an explicit retry may replace the failed result. Failure to resolve a shared input such as the selected Image Asset instead affects each Stage that declares that input because none may silently substitute different source data.

Preparation progress and errors are retained per Stage identity. Different Stage generators have different units, phases, and completion semantics, so Combined may summarize which Stages are preparing or failed but does not manufacture one Sequence-wide completion percentage.

Background preparation scheduling is unrelated to either physical Sequence order or Combined painter order. Demanded Stage identities prepare independently; version one may run existing Ink and supporting-Stage workers concurrently, while later concurrency limits and prioritization remain Harness policy that cannot alter Stage identity or deterministic results.

Demand starts preparation but a temporary view change does not cancel current work. Once requested, a Stage job may finish into retained state unless a newer preparation identity supersedes it, the artist explicitly retries or cancels it, the owning Sketch changes, or the session is disposed. This prevents rapid isolated/Combined switching from repeatedly throwing away the same expensive work.

The Studio may deepen its existing repeated one-worker-per-job transport lifecycle when Stage preparation introduces another background consumer. That shared module is deliberately narrow: worker construction, listener ownership, cancellation, disposal, failure normalization, stale-callback rejection, and best-effort exactly-once termination. Plot Stage protocols and runtimes, progress interpretation, result validation, retained session state, and the Stage-identity scheduler remain domain-specific; existing Ink Shading preparation adapts its retained result into the Primary Stage instead of being rewritten through a general worker framework.

Authored Sequence order means physical execution and ordered export identity, not an immutable painter order for every presentation. Combined preview stacking is a separate transient input to Scene presentation. Version one exposes Watercolor Forms above Ink so the supporting geometry remains visible for registered inspection, while Combined composition accepts an explicit ordered collection of Stage identities rather than deriving painter order irreversibly from declaration position. Reordering a preview never changes Stage identity, physical execution order, Presets, or exported geometry.

Geometry-changing cross-Stage occlusion, if later authored, is a third relationship rather than another meaning for either order. It composes unfinalized Stage Scenes explicitly before the existing Hidden-line pass. Version one pays only to retain those ordinary source Scenes at the already-planned Stage preparation boundary; it adds no occlusion graph, persisted occlusion state, cross-Stage pass, or Stage dependency protocol. Page finalization and SVG serialization remain downstream, so introducing an authored occlusion composition later extends the middle of the pipeline instead of replacing Stage generation or output.

Stage color is artifact styling and an operator cue, not physical Plot Tool truth. Version one persists one color per Stage and uses it to style that Stage in isolated and Combined Studio presentation and its SVG paths, while the operator remains responsible for loading the intended physical pencil or pen. Ink defaults to its established black and Watercolor Forms defaults to bright pink so Combined is legible immediately and older Presets receive deterministic values. Color finalization reuses retained Stage geometry just like physical Tool-width styling and never reruns its Stage generator. A separate preview-only tint and a purported machine-controlled tool color would add two states without matching the actual plotter workflow, so neither is introduced.

The Sequence shares physical paper dimensions and margins while each Stage owns its Stage color, Tool width, and optional final-Page outline. Version one exports separate ordered plotter SVGs with identical page extent and coordinate mapping; a later portable layered SVG may package those same Stage artifacts without changing their generation model. Physical paper repositioning and distortion remain operator-owned.

We rejected a Photo Scribble-only hard-coded workflow because Seed-dependent geometry Sketches such as Grass Hills and Leaf Field are plausible next consumers. We deferred a user-assembled general layer editor because no current workflow yet justifies exposing arbitrary Stage creation, duplication, or reordering, and rejected Primitive layer tags as the primary model because artifact-internal pen routing does not express independently generated, separately executed plot assets. Photo Scribble is the sole first-version Sequence consumer; Watercolor Forms is its sole first-version supporting Stage and reuses the same headless generator and complete seven-control surface as the independent Watercolor Forms Sketch, with independent Stage-owned values. Pencil Contour, Flowing Contours, and Luminance Isophotes remain candidate generators for later curated Stage instances rather than alternatives the architecture must discard.
