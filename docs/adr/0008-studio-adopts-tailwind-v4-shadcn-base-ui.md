# Studio adopts Tailwind v4 + shadcn (Base UI) as its component foundation

The studio's UI redesign (slice #152) needs a component vocabulary — sidebar, canvas frame, control rows, popovers — that many downstream tasks build against in parallel. We adopt **Tailwind v4** (via the `@tailwindcss/vite` plugin) for styling and **shadcn/ui built on Base UI primitives** (`@base-ui-components/react`, composed with `class-variance-authority` + `clsx` + `tailwind-merge`, iconography from `lucide-react`) as the component layer. shadcn is copy-in source, not a runtime dependency: components land in `apps/studio/src/components/ui/*` and are ours to edit, so the "framework" is a starting point we own rather than a version we track. Styling is token-first — a single design-token layer in `src/index.css` (`--background`, `--foreground`, `--border`, `--muted`, …) that every component references through Tailwind utilities (`bg-card`, `text-muted-foreground`, `border`), so the palette is themed in one place, not scattered across components.

We considered **plain hand-written CSS** (what the studio shell shipped with): it has zero dependencies and no supply-chain surface, but every downstream #152 task would hand-roll its own layout/state/focus/keyboard-interaction primitives, and a shared look would be a convention rather than a mechanism — exactly the drift the redesign is trying to avoid. We considered **Base Web** (Uber's component kit): batteries-included, but it owns your styling engine (Styletron) and its opinionated look is hard to bend to a bespoke generative-art studio, whereas the shadcn model hands us unstyled Base UI behavior + Tailwind tokens we fully control. The deciding factor is that #152 fans out into many parallel control/canvas/sidebar tasks: a copy-in, token-driven kit lets them share accessible primitives and one palette without coupling to an upstream component release.

**Scope guard: a single, dark, neutral theme only.** `src/index.css` defines the dark palette directly on `:root` (with `color-scheme: dark`); there is deliberately **no** light-theme variant, no `.dark` class, and no theme toggle — those are explicitly out of scope for this slice. A future theming decision, if one is ever wanted, gets its own ADR.

## The supply-chain-lockdown implication

Tailwind v4 moved its engine to a native Rust core (`@tailwindcss/oxide`), which is the kind of dependency this machine's npm/pnpm lockdown exists to scrutinize (see `docs/agents/locked-down-npm.md`): lifecycle build scripts are gated by `strictDepBuilds` + `allowBuilds`, packages younger than the `min-release-age` window are refused, and exotic subdeps are blocked. Adopting Tailwind therefore had to clear that gate on the prebuilt-binary path, the same way the already-tolerated `esbuild` does — not by weakening the gate.

It cleared cleanly. `@tailwindcss/oxide` ships per-platform prebuilt binaries as optional dependencies (`@tailwindcss/oxide-darwin-arm64`, etc.), and its only npm `build` script is a non-lifecycle entry (not `postinstall`/`install`/`prepare`), so pnpm never flags it: `pnpm install` exits 0 with **no** `ERR_PNPM_IGNORED_BUILDS` and **no** new `allowBuilds` entry required. Versions were pinned under the age window (Tailwind 4.3.1, `lucide-react` 1.21.0) rather than to `latest`, which sat inside the refusal window at the time.

## Consequences

- **The escape hatch, unused but reserved.** Had oxide needed its build script, the sanctioned move would have been a single reviewed, committed entry in `pnpm-workspace.yaml` — `allowBuilds: { '@tailwindcss/oxide': true }` — per-repo and auditable in the diff, exactly as `esbuild: false` already records a reviewed decision there. It was not needed, so `pnpm-workspace.yaml` is unchanged. The forbidden moves remain forbidden: no `dangerouslyAllowAllBuilds`, no lowering `min-release-age`, no editing `~/.npmrc` or the global pnpm rc.
- **Verify with package-local binaries.** A fresh worktree's `pnpm install` may still exit 1 on the pre-existing `esbuild` ignored-build nag (benign). Typecheck and test the studio with `apps/studio/node_modules/.bin/tsc --noEmit` and `apps/studio/node_modules/.bin/vitest run`, and prove the CSS compiles with `apps/studio/node_modules/.bin/vite build` (the oxide prebuilt binary runs there) — not `pnpm run <script>`, whose pre-flight re-surfaces the nag.
- **No behavior change in this slice.** This is tooling groundwork: the existing studio shell (live params, seed, randomize, presets, PNG/SVG export, transport) renders and functions exactly as before. The only visible addition is one shadcn/Tailwind-styled proof element on a dark surface; `packages/core` (the Sketch contract, Scene IR, renderers, determinism spine) is untouched.
- **shadcn components are owned source.** Because they are copied in, upgrades are manual re-copies, not `pnpm update`. That is the trade for full control over markup and tokens; the underlying Base UI primitive package (`@base-ui-components/react`) is the only tracked runtime dependency, and it was pinned at its release candidate (`1.0.0-rc.0`, the only published line).
